# ch11: Skills 系统规格说明

## 1. 背景与目标

CN Code 已经具备 Slash 命令、上下文压缩、记忆、会话持久化和工具调用能力。当前 Slash 命令主要依赖 Java 代码内置注册，适合确定性的本地动作，但不适合承载可扩展的复杂工作流。用户需要一种可分发、可热更新、可按需激活的能力包，让 CN Code 能在不同项目里复用 SOP、工具白名单、执行模式和专属工具。

本章目标是交付一套 Skill 系统：

1. Skill 用声明式元信息加 Markdown 正文描述，正文作为发给模型的 SOP 指令。
2. Skill 按项目、用户、内置三层发现，同名按优先级覆盖。
3. 启动时只注入 Skill 名称和一句说明，完整 SOP 在需要时按需加载。
4. 激活后的 Skill 指令进入环境上下文，而不是混进普通消息历史，并支持多个 Skill 并存。
5. Skill 可选择共享当前对话执行，或隔离成独立对话执行后摘要回流。
6. Skill 可收窄可见工具集，并在白名单引用不存在工具时 fail-fast。
7. Skill 可作为 Slash 命令调用，并提供管理命令查看、详情和重扫。
8. 提供内置 commit / review / test 三个样板 Skill，既可直接使用，也可作为模板参考。

本章不实现 Skill 市场、远程分发和版本管理。

## 2. 目标用户

1. 希望在项目中沉淀固定工作流的 CN Code 使用者。
2. 希望把“提交、审查、测试、排障”等重复任务包装成可复用能力的开发者。
3. 希望用工具白名单降低模型误用工具概率、落实最小权限的用户。
4. 希望通过目录型能力包扩展 CN Code 专属工具的高级用户。

## 3. 功能需求

### F1. Skill 描述格式

Skill 必须支持“YAML frontmatter + Markdown 正文”的入口文件格式。frontmatter 用于描述唯一名字、一句话说明、适用场景、可见工具白名单、执行模式、所用模型和上下文携带策略；Markdown 正文作为完整 SOP 指令。

Skill 也可以支持“元信息文件 + prompt 正文文件”的目录型布局，方便把元信息、SOP、工具声明和工具实现拆分维护。

### F2. Skill 目录层级与覆盖

系统必须按固定层级发现 Skill：内置层、用户层、项目层。项目层优先级最高，用户层次之，内置层最低。同名 Skill 后加载者覆盖先加载者。

单个 Skill 解析失败时必须跳过该 Skill 并记录可诊断日志，不得阻断其他 Skill 加载。

### F3. 两阶段加载

启动阶段只加载 Skill 的名称、一句话说明、来源和必要元信息，用于构建 Skill 摘要并注入给 Agent。

完整 SOP 正文、目录型 Skill 的附加资源和专属工具只在 Skill 被调用或查看详情时加载。源文件变更后，下次调用应重新读取，以支持热更新。

### F4. Skill 摘要注入

启动后，系统必须把可用 Skill 的名字和一句说明注入模型上下文，让 Agent 知道可以按需使用哪些能力。

摘要不得包含完整 SOP 正文，避免启动上下文膨胀。

### F5. Skill 激活工具

系统必须提供一个内置工具用于加载指定 Skill 的完整指令和专属工具。该工具属于系统级工具，不受当前 Skill 的工具白名单限制，避免白名单把后续 Skill 嵌套调用入口锁死。

当 Agent 判断需要某个 Skill 时，应通过该工具加载完整 Skill，而不是依赖启动摘要里的简略说明脑补 SOP。

### F6. 激活 Skill 的上下文位置

激活后的完整 Skill 指令必须进入环境上下文或等价的高优先级稳定上下文，而不是作为普通用户消息或助手消息长期混在消息历史里。

每轮 Agent Loop 重新构建请求时，已激活 Skill 都应位于显眼位置。多个 Skill 同时激活时，各自指令必须并存，并能在清空会话时被一起清除。

### F7. 共享执行模式

Skill 必须支持共享当前对话上下文的执行模式。该模式下，Skill SOP 在主会话中生效，执行结果保留在主对话历史里。

共享模式适合 commit、review、test 等需要直接基于当前工作上下文继续行动的工作流。

### F8. 隔离执行模式

Skill 必须支持隔离执行模式。该模式下，系统为 Skill 开启独立对话执行，完成后把结果摘要回流到主对话。

隔离模式必须支持三档父上下文携带策略：完全清空、最近若干条、全量摘要。选择哪一档由 Skill 元信息声明。

### F9. 模型选择

Skill 可以声明期望模型。未声明时继承当前会话模型。系统不要求本章实现复杂模型路由策略，但必须保留元信息并在执行链路中可观测。

当声明模型不可用时，系统应给出可见提示或回退到当前模型，具体策略必须一致且可测试。

### F10. 工具白名单

Skill 可以声明可见工具白名单，用于收窄当前可用工具集。白名单为空或未声明时表示不额外收窄。

启动扫描或注册阶段如果发现白名单引用不存在的工具，必须 fail-fast 暴露配置错误，不能等到执行中途才失败。

Skill 加载工具本身不受白名单约束。

### F11. 目录型 Skill

系统必须支持目录型 Skill。目录型 Skill 至少包含入口说明文件；可选包含自己的工具 schema 和工具实现脚本。

目录型 Skill 应作为一个可分发能力包加载。专属工具只在 Skill 激活后进入当前会话可用工具集，未激活时不污染全局工具列表。

### F12. Slash 命令注册

Skill 加载后必须自动注册成 `/<skill-name>` Slash 命令，并出现在帮助或 Skill 列表中。

用户执行 `/<skill-name>` 时，系统应重新读取源文件以支持热更新，并按 Skill 声明的执行模式启动共享或隔离执行。

### F13. Skill 管理命令

系统必须提供 Skill 管理入口，用于查看已发现 Skill 列表、查看单个 Skill 详情、强制重新扫描 Skill 目录。

管理命令返回本地系统消息，不应被当作普通用户消息发送给模型。

### F14. 清空会话联动

清空对话的命令必须同时清空当前已激活 Skill 列表和 Skill 引入的临时工具过滤状态，避免新对话残留上一轮 SOP。

清空只影响当前会话运行态，不删除磁盘上的 Skill 文件。

### F15. 内置样板 Skill

系统必须内置 commit、review、test 三个 Skill 样板。

内置样板应覆盖共享模式和隔离模式两类典型用法，并能作为用户自定义 Skill 的参考模板。项目层或用户层同名 Skill 可以覆盖内置样板。

## 4. 非功能要求

### N1. 启动性能

启动阶段不得读取大段 SOP 正文或专属工具实现内容。Skill 摘要加载应保持轻量，避免明显拖慢 CN Code 启动。

### N2. 容错边界

目录不存在、单个 Skill 解析失败、单个 Skill 文件缺失正文时，不得导致整个 Skill 系统不可用。只有工具白名单引用不存在工具这类会影响安全和执行正确性的错误需要 fail-fast。

### N3. 安全边界

Skill 文件、工具 schema 和工具实现脚本的加载必须限制在 Skill 所在目录或 CN Code 允许的数据目录内。Skill 内容本身视为不可信输入，不能绕过现有权限、沙箱和工具安全规则。

### N4. 最小权限

Skill 工具白名单必须以收窄能力为目标，不得扩大当前会话权限。白名单过滤后的工具集应与现有权限检查共同生效。

### N5. 热更新可预测

执行 Skill 时重新读取源文件，但读取失败时不得用半成品覆盖已可用的旧缓存。用户应能通过管理命令触发重扫并看到错误提示。

### N6. 上下文可控

Skill 完整 SOP 不得无界追加到普通历史里。激活 Skill 的环境上下文需要可清除、可重建、可被上下文压缩恢复块引用。

### N7. 可测试性

Skill 解析、层级覆盖、白名单校验、共享执行、隔离执行、Slash 注册和清空联动都必须能通过自动化测试或明确手动验收验证。

## 5. 设计骨架

本章拆成六个能力边界：

1. Skill 编目：负责扫描三层目录、解析元信息、维护来源和覆盖关系、按需读取完整正文。
2. Skill 激活状态：负责记录当前会话已激活 Skill、构建环境上下文、支持清空和压缩恢复。
3. Skill 执行器：负责按共享或隔离模式执行 Skill，处理参数、模型声明和父上下文携带策略。
4. 工具过滤与专属工具：负责白名单校验、工具可见性收窄、目录型 Skill 专属工具加载。
5. Slash 与管理入口：负责自动注册 `/<skill-name>`、列表/详情/重扫、清空会话联动。
6. 内置样板：负责提供 commit / review / test 三个默认 Skill，并允许用户或项目覆盖。

主流程：

1. 启动时扫描内置、用户、项目三层 Skill。
2. 校验轻量元信息和工具白名单，生成 Skill 摘要。
3. 把 Skill 摘要注入模型上下文。
4. 用户或 Agent 触发 Skill 加载。
5. 系统按需读取完整 SOP 与专属资源。
6. 共享模式把 SOP 钉入当前环境上下文；隔离模式创建独立执行上下文并摘要回流。
7. Slash 命令和管理命令提供可见操作入口。
8. 清空会话时清掉已激活 Skill 运行态。

## 6. Out of Scope

1. 不实现 Skill 市场。
2. 不实现远程 Skill 分发。
3. 不实现 Skill 版本管理。
4. 不实现签名校验、可信发布者和依赖锁定。
5. 不实现文件 watcher 自动实时监听新增 Skill；新增 Skill 可通过重扫或重启发现。
6. 不实现复杂的 Skill 编排、pipeline 和多 Skill 工作流语言。
7. 不实现跨项目共享运行态；用户层 Skill 文件可共享，但激活状态只属于当前会话。
8. 不实现专属工具脚本的权限升级；专属工具仍受现有权限系统约束。

## 7. 验收标准

1. 在项目、用户、内置三层存在同名 Skill 时，项目层覆盖用户层，用户层覆盖内置层。
2. 启动注入的 Skill 摘要只包含名称和一句说明，不包含完整 SOP 正文。
3. 执行某个 Skill 时，系统重新读取该 Skill 源文件并加载完整 SOP。
4. 激活后的 Skill 出现在环境上下文或等价高优先级上下文中，而不是作为普通历史消息长期追加。
5. 同时激活两个 Skill 后，两者 SOP 都能在后续 Agent Loop 中被构建进上下文。
6. `/clear` 或等价清空命令执行后，已激活 Skill 列表为空。
7. Skill 声明不存在的工具白名单项时，启动或重扫阶段出现 fail-fast 错误。
8. Skill 加载工具在任意 Skill 白名单下仍可被系统使用。
9. 目录型 Skill 的入口文件、工具 schema 和工具实现脚本能作为同一个能力包被发现。
10. `/skills` 能列出已发现 Skill，详情命令能显示单个 Skill 的来源、模式、模型、白名单和说明。
11. 重扫命令能发现新增 Skill 或反映解析错误。
12. `/commit`、`/review`、`/test` 三个内置样板 Skill 可见；项目或用户同名 Skill 可以覆盖它们。
13. 共享模式 Skill 的执行结果留在主会话历史。
14. 隔离模式 Skill 的执行结果以摘要形式回流主会话。
15. 隔离模式的三档上下文携带策略都可观测验证。
16. 单个坏 Skill 不影响其他有效 Skill 加载。
17. `.\gradlew.bat smokeTest` 通过。
