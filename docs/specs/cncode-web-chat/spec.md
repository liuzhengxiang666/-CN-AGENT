# CN Code Web 聊天界面规格说明

## 1. 背景与目标

当前 CN Code 已具备基础 LLM 对话链路：能够读取 YAML 配置，选择 Provider，调用大模型 API，流式输出回复，并在 Java 进程内保存多轮会话历史。终端 TUI 在不同 Windows 终端中的兼容性和视觉表现不够稳定，因此本阶段改为提供一个本地 Web 聊天界面。

用户通过命令启动 CN Code Web 模式后，Java 后端启动本地 HTTP 服务，默认从端口 `8765` 开始寻找可用端口，并自动打开浏览器访问该地址。浏览器页面展示一个稳定的聊天界面，包含聊天区、输入区和侧边栏。用户输入问题后，前端调用 Java 后端接口，后端使用当前 YAML 配置调用大模型 API，并通过 SSE 将模型回复流式推送给前端。前端应逐字或逐块显示模型回复。

同一个 Java 后端进程内需要保存多轮会话历史。后端进程退出后，历史不需要持久化保存。

## 2. 功能需求

### F1. Web 模式启动

用户可以通过启动参数进入 Web UI 模式，例如：

```powershell
cncode --web
```

启动后，Java 后端启动本地 HTTP 服务。

### F2. 自动端口选择

Web 服务默认尝试监听 `8765`。

如果 `8765` 被占用，自动尝试 `8766`、`8767` 等后续端口，直到找到可用端口。

启动成功后，终端打印实际访问地址。

### F3. 自动打开浏览器

Web 服务启动成功后，程序自动打开默认浏览器访问本地 Web UI 地址。

如果自动打开浏览器失败，程序仍保持服务运行，并在终端提示用户手动访问地址。

### F4. 前端页面布局

Web UI 页面包含：

- 聊天区：显示用户消息、助手消息和系统提示。
- 输入区：用于输入用户问题。
- 侧边栏：显示 CN Code 标题、当前 `protocol/model`、服务状态、会话消息数量和后续工具区占位。

### F5. 输入行为

用户在输入框中按 Enter 发送当前消息。

用户按 Shift+Enter 时，在输入框中插入换行。

发送成功后输入框清空。

### F6. 流式回复

前端发送消息后，后端调用当前配置的 Provider。

后端通过 SSE 将模型回复片段推送给前端。

前端收到片段后，逐字或逐块追加显示到当前助手消息中。

### F7. 多轮会话历史

Java 后端在内存中保存当前会话历史。

后续请求调用 Provider 时必须携带之前的用户消息和助手消息。

后端进程退出后，本阶段不保存历史。

### F8. 状态信息

前端侧边栏显示：

- 当前 provider 协议
- 当前模型名
- 服务连接状态
- 当前会话消息数量
- 工具区占位：`Tools: not enabled`

### F9. 错误显示

当 Provider 调用失败、SSE 断开、网络错误或后端异常时，前端应显示清晰错误提示。

错误发生后，用户仍可以继续发送新消息。

### F10. 保留原有模式

本阶段不删除 REPL 或 TUI。

`--web` 进入 Web UI 模式。

不带参数时，保持现有默认行为。

## 3. 非功能需求

### N1. 实现语言与技术栈

后端继续使用 Java 21。

前端使用纯 HTML、CSS 和 JavaScript，不引入 React、Vue、Vite 等前端构建工具。

### N2. 本地优先

Web 服务只监听本机地址，例如 `127.0.0.1`。

本阶段不提供公网访问能力。

### N3. 流式体验

模型回复应尽快显示在浏览器聊天区。

不得等待完整回复生成后才一次性显示。

### N4. Provider 解耦

Web 后端不得直接依赖 OpenAI、Anthropic 或 DeepSeek 的具体请求格式。

后端仍通过已有 `ChatProvider` 抽象调用模型。

### N5. 会话一致性

Web 模式和现有 REPL/TUI 模式使用同一套 `ChatSession` 会话模型。

同一后端进程内，消息历史按顺序保存。

### N6. UI 体验

Web UI 应比终端 TUI 更稳定、更清晰。

界面应具备聊天产品的基本质感：清晰的消息层级、合理留白、可读的输入区和侧边栏状态信息。

### N7. 依赖控制

本阶段不引入前端构建系统。

后端优先使用 Java 标准库 HTTP Server，除非现有项目已经有合适 Web 框架。

### N8. 安全边界

服务默认仅供本机使用。

API Key 只保存在后端配置中，不应暴露到前端页面或浏览器 JavaScript 中。

### N9. 可验收性

开发完成后，应能启动 Web 模式，自动打开浏览器，输入真实问题，看到模型回复流式显示，并验证多轮上下文有效。

## 4. 不做的事

本阶段明确不实现以下能力：

1. 不实现 tool calling。
2. 不实现文件读取、搜索、编辑或删除。
3. 不实现代码修改、补丁生成或自动重构。
4. 不实现用户登录、鉴权或多用户。
5. 不实现公网访问。
6. 不持久化保存会话历史。
7. 不新增前端构建系统。
8. 不引入 React、Vue 或 Vite。
9. 不实现多会话管理。
10. 不实现模型切换 UI。
11. 不实现配置文件编辑 UI。
12. 不在前端暴露 API Key。
13. 不删除现有 REPL/TUI 模式。

## 5. 验收标准

### A1. Web 模式启动验收

执行 `cncode --web` 后，Java 后端启动本地 HTTP 服务。

服务监听地址为 `127.0.0.1`。

### A2. 端口选择验收

当 `8765` 可用时，服务使用 `8765`。

当 `8765` 被占用时，服务自动尝试后续端口，例如 `8766`。

终端打印实际访问地址。

### A3. 浏览器打开验收

Web 服务启动成功后，程序尝试自动打开默认浏览器访问 Web UI。

如果自动打开失败，终端显示可手动访问的 URL。

### A4. 页面布局验收

浏览器页面包含聊天区、输入区和侧边栏。

侧边栏显示 CN Code 标题、当前 `protocol/model`、服务状态、消息数量和 `Tools: not enabled`。

### A5. 输入验收

用户在输入框输入一行文本后按 Enter，消息被发送。

用户按 Shift+Enter 时，输入框插入换行，不发送消息。

发送后输入框清空。

### A6. 流式回复验收

用户发送真实问题后，前端显示助手消息占位。

模型回复片段通过 SSE 逐字或逐块显示到助手消息中。

回复不等待完整生成后才显示。

### A7. 多轮上下文验收

用户发送：“我叫小明，喜欢 Java。”

模型回复完成后，继续发送：“我刚才说我喜欢什么语言？”

模型回复能参考前文，并指出用户喜欢 Java。

### A8. 错误恢复验收

Provider 调用失败时，前端显示清晰错误提示。

错误后用户仍可以继续发送新消息。

### A9. 安全验收

浏览器页面源码和 JavaScript 不包含 API Key。

前端只知道 provider/model，不知道密钥。

### A10. 回归验收

不带参数启动时，现有默认模式仍可用。

`--repl` 仍进入基础 REPL。

`--tui` 仍尝试进入 TUI。
