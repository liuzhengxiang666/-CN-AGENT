# CN Code 单工具 Agent 能力规格说明

## 1. 背景与目标

CN Code 当前已经具备 Web/REPL/TUI 对话能力，可以读取 YAML 配置，调用当前大模型 Provider，并将模型回复流式展示给用户。但当前模型只能基于已有上下文生成文本，不能实际读取项目文件、搜索代码、修改文件或执行命令。

本阶段目标是给 CN Code 增加第一版工具系统，使它从纯聊天助手升级为能执行本地任务的单工具 Agent。用户提出需求后，模型可以在一次回复中请求调用一个工具。CN Code 解析模型的工具调用请求，执行对应本地工具，将结构化执行结果写回对话历史，再请求模型基于工具结果生成最终回复。

本阶段每次用户请求最多允许执行一个工具。工具执行完成并回灌结果后，模型必须生成最终文本回复，不能继续发起第二个工具调用。这样先跑通工具系统、工具注册、工具调用解析、工具结果回灌和错误反馈链路，为后续多工具 Agent Loop 打基础。

第一版工具系统包含六个核心工具：读文件、写文件、改文件、执行命令、按模式找文件、搜索代码内容。

本阶段参考工具系统模板中的 Java ch03 核心设计，但只落地必要骨架：统一 Tool 接口、工具注册中心、结构化 ToolResult、六个核心工具、OpenAI-compatible 工具 schema 与流式 tool call 解析。不引入 ch04 Agent Loop，也不引入 ToolSearch、AskUser、Deferred 工具披露、并发工具批次等高级能力。

## 2. 功能需求

### F1. 统一 Tool 接口

系统提供统一的 Tool 抽象。

每个工具都必须提供：

- 工具名称
- 工具描述
- 参数 JSON Schema
- 执行方法

工具描述必须明确说明“什么时候应该使用该工具”，避免模型在需要读取文件、搜索代码或执行命令时凭空猜测。

工具执行方法接收结构化参数，并返回结构化结果。

### F1a. 工具分类

系统为工具提供轻量分类：

- `READ`：只读工具，例如读文件、找文件、搜索代码。
- `WRITE`：写入工具，例如写文件、改文件。
- `COMMAND`：命令工具，例如执行命令。

本阶段分类只作为元信息和后续权限、并发、危险命令确认的基础，不实现并行调度和权限审批。

### F2. 工具注册中心

系统提供工具注册中心。

注册中心负责：

- 集中登记所有工具
- 按名称查找工具
- 输出当前可用工具列表
- 将工具元信息转换成 LLM Provider 能识别的工具定义格式

注册中心应保持注册顺序稳定，默认注册顺序为：

1. `read_file`
2. `write_file`
3. `replace_file`
4. `run_command`
5. `find_files`
6. `search_code`

当 Provider 为 OpenAI-compatible 时，注册中心输出 OpenAI tools/function 形状；内部工具元信息不绑定具体 Provider。

### F3. 六个核心工具

第一版实现六个工具：

1. 读文件：读取指定路径文件内容。
2. 写文件：向指定路径写入完整内容。
3. 改文件：使用原文唯一匹配替换方式修改文件。
4. 执行命令：执行 shell 命令并返回 stdout、stderr、退出码。
5. 按模式找文件：按 glob 或文件名模式查找文件。
6. 搜索代码内容：按文本或正则搜索项目文件内容。

### F4. 工具执行结果

工具执行必须返回结构化结果。

结果至少包含：

- 是否成功
- 工具名称
- 可读摘要
- 详细输出
- 错误信息

工具失败时，失败信息也必须作为结构化结果返回给模型，而不是导致程序崩溃。

### F5. 超时控制

工具执行必须带超时。

当工具执行超过限制时，系统中断或停止等待，并返回结构化超时错误结果。

### F6. 改文件唯一匹配

改文件工具使用“原文唯一匹配替换”。

如果原文匹配 0 次，返回清晰错误。

如果原文匹配多次，返回清晰错误。

只有原文恰好匹配 1 次时，才执行替换写入。

### F7. 工具调用解析

LLM 客户端必须能解析模型流式输出中的工具调用。

当模型流式返回工具名和 JSON 参数碎片时，系统应能拼接完整参数。

参数拼接完成后，系统按工具名查找并执行对应工具。

### F8. 工具结果回灌

工具执行完成后，系统将工具执行结果写回对话历史。

随后系统再次调用模型，让模型基于工具结果生成最终回复。

### F9. 单工具限制

每次用户请求最多执行一个工具。

工具结果回灌后，第二次模型调用不允许继续发起工具调用。

如果模型继续请求工具，系统应拒绝并提示本阶段只支持单工具调用。

### F10. Provider 集成

工具调用能力优先接入当前 OpenAI-compatible Provider，以支持 DeepSeek。

Anthropic Provider 可以保留扩展点，但本阶段不要求完整支持 Anthropic tool use。

### F11. UI 展示

Web UI 至少能显示：

- 模型准备调用哪个工具
- 工具执行成功或失败
- 工具执行摘要
- 模型基于工具结果生成的最终回复

REPL/TUI 可以先用文本方式显示工具过程。

## 3. 非功能需求

### N1. 实现语言与运行环境

继续使用 Java 21 实现。

工具系统应与现有 Provider、ChatSession、Web UI 结构兼容。

### N2. 安全边界

工具默认只能在项目工作目录内读写和搜索文件。

禁止通过相对路径逃逸到工作目录之外。

执行命令默认在项目工作目录内执行。

### N3. 可诊断性

工具失败时，错误结果必须清晰说明失败原因。

例如：

- 文件不存在
- 路径越界
- 匹配不到原文
- 原文匹配多次
- 命令超时
- 命令退出码非 0

### N4. Provider 解耦

工具接口、注册中心和工具执行逻辑不得绑定具体 LLM Provider。

Provider 只负责把模型协议中的 tool call 转换为内部统一工具调用结构。

### N5. 单工具 MVP

本阶段优先跑通单工具完整链路。

不为多工具循环、复杂规划、并行工具执行做过度设计。

### N6. 结果可控

工具结果返回给模型时应限制长度。

过长的文件内容、搜索结果或命令输出需要截断，并在结果中说明已截断。

### N7. 用户可见性

用户应该能看到工具执行过程，而不是只看到最终答案。

Web UI 中工具过程应作为可读状态或系统消息显示。

### N8. 兼容现有对话模式

新增工具系统后，普通纯聊天仍应可用。

当模型不需要工具时，对话流程应与之前保持一致。

## 4. 不做的事

本阶段明确不实现以下能力：

1. 不实现多工具连环调用。
2. 不实现一次请求调用多个工具。
3. 不实现并行工具执行。
4. 不实现长期任务后台运行。
5. 不实现用户确认弹窗或权限审批 UI。
6. 不实现 Git 专用工具。
7. 不实现网络请求工具。
8. 不实现浏览器控制工具。
9. 不实现图片、PDF、Word 等非文本文件专用工具。
10. 不实现跨工作目录读写文件。
11. 不实现持久化工具调用历史。
12. 不要求 Anthropic Provider 完整支持工具调用。
13. 不实现复杂 Agent Planner。
14. 不实现自动重试工具调用，模型可根据错误自行重试到下一轮用户请求。
15. 不实现 ToolSearch 工具。
16. 不实现 AskUser 工具。
17. 不实现 Deferred 工具发现机制。
18. 不实现按工具分类的并行执行策略。
19. 不实现权限系统、Hook 系统、SubAgent、Team 或 Worktree 工具。

## 5. 验收标准

### A1. 工具注册验收

系统启动后，六个核心工具都已注册。

注册中心能按名称查找到每个工具。

注册中心能输出 LLM Provider 可用的工具定义列表。

工具定义列表中，每个工具都有明确描述和参数 Schema。

OpenAI-compatible 输出中，每个工具都转换为 API 可识别的 function/tool 格式。

### A2. 文件读取验收

用户请求“读取 AGENTS.md 的内容”时，模型能调用读文件工具。

工具读取项目内 `AGENTS.md`，并将内容返回给模型。

模型最终回复中能基于文件内容回答。

### A3. 文件写入验收

用户请求创建一个测试文本文件时，模型能调用写文件工具。

工具在项目工作目录内写入指定文件。

写入成功后，模型最终回复说明已完成。

### A4. 文件修改验收

用户请求把测试文件中的唯一原文替换为新内容时，模型能调用改文件工具。

原文匹配 1 次时修改成功。

原文匹配 0 次或多次时返回结构化错误，模型最终回复能解释原因。

### A5. 命令执行验收

用户请求执行一个简单命令时，模型能调用执行命令工具。

工具返回 stdout、stderr 和退出码。

命令超时时返回结构化超时错误。

### A6. 找文件验收

用户请求查找某类文件时，模型能调用按模式找文件工具。

工具返回匹配路径列表。

结果过长时会截断并说明。

### A7. 搜索代码验收

用户请求搜索某个关键字时，模型能调用搜索代码内容工具。

工具返回匹配文件、行号和文本片段。

结果过长时会截断并说明。

### A8. 路径安全验收

当模型请求读取或写入工作目录外路径时，工具拒绝执行。

结果中说明路径越界。

### A9. 单工具限制验收

同一次用户请求中，系统最多执行一个工具。

工具结果回灌后，如果模型继续请求工具，系统拒绝并提示本阶段只支持单工具调用。

### A10. Web UI 工具过程验收

在 Web UI 中发起需要工具的请求时，页面能显示工具调用过程和执行结果摘要。

最终回复显示在助手消息中。

### A11. 纯聊天回归验收

用户发送不需要工具的问题时，模型可以直接回复，不触发工具执行。

### A12. 构建验收

`.\gradlew.bat check` 通过。

`.\gradlew.bat installDist` 通过。
