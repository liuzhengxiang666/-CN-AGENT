# CN Code TUI 对话界面规格说明

## 1. 背景与目标

CN Code 是一个使用 Java 21 开发的终端 AI 编程助手。当前已有基础 REPL 对话能力，支持读取 YAML 配置、选择 Provider、调用大模型 API、流式输出回复，并在当前进程内保存多轮对话历史。

本阶段目标是将基础 REPL 升级为简洁 Claude Code 风格的终端 TUI 对话界面。用户启动 CN Code 后，应进入一个结构化终端界面：上方为历史对话区域，下方为多行输入框，底部为状态栏。用户可以在输入框中输入问题，按 Ctrl+Enter 发送；CN Code 调用配置的大模型 API，并将模型回复以流式方式追加到历史对话区域。

同一次运行期间，CN Code 必须继续保存多轮上下文，使模型能参考之前的对话继续回答。历史对话区域需要支持鼠标滚轮滚动；模型回复流式输出时，界面自动滚动到底部。若当前终端不支持 TUI 或 TUI 初始化失败，程序应退回到现有基础 REPL，保证仍可进行基本对话。

## 2. 功能需求

### F1. TUI 启动

用户执行 `cncode` 后，程序优先尝试启动 TUI 对话界面。

TUI 启动成功后，用户应看到一个结构化终端界面，包含历史对话区域、多行输入框和底部状态栏。

### F2. 历史对话区域

TUI 上方显示历史对话区域。

历史对话区域应按时间顺序显示用户消息和助手消息。

用户消息和助手消息应能被区分，例如使用不同前缀、颜色或标记。

### F3. 多行输入框

TUI 下方显示多行输入框。

用户可以输入多行内容。

按 Enter 时，输入框插入换行。

按 Ctrl+Enter 时，发送当前输入内容。

发送后，输入框清空。

### F4. 退出命令

当用户发送的完整输入内容为 `exit` 或 `quit` 时，程序正常退出。

退出时不需要保存会话历史。

### F5. 流式回复显示

用户发送问题后，CN Code 调用当前配置的 Provider。

模型回复到达时，TUI 应将回复内容以流式方式追加到当前助手消息中。

程序不得等待完整回复结束后才一次性刷新界面。

### F6. 自动滚动到底部

模型回复流式输出时，历史对话区域自动滚动到底部，使用户能持续看到最新回复。

### F7. 鼠标滚轮滚动

历史对话区域支持鼠标滚轮滚动。

用户可以通过鼠标滚轮查看较早的历史消息。

### F8. 多轮上下文记忆

CN Code 在当前进程内继续保存用户消息和助手消息历史。

后续问题调用 Provider 时，必须携带前文历史，使模型能参考之前的对话。

程序退出后，本阶段不保存历史。

### F9. 状态栏

底部状态栏显示当前配置名、Provider 协议和模型名。

第一版配置名固定显示为 `default`。

状态栏格式示例：

```text
default    openai/deepseek-v4-flash
```

### F10. REPL fallback

如果 TUI 初始化失败、终端能力不足或 Lanterna 无法启动，程序应打印清晰提示，并自动退回现有基础 REPL。

fallback 后仍应支持输入问题、流式回复、多轮上下文和 `exit` / `quit` 退出。

## 3. 非功能需求

### N1. 实现语言与运行环境

继续使用 Java 21 实现。

TUI 实现基于 Lanterna。

### N2. 中文交互约定

终端界面、错误提示、状态说明和代码注释优先使用中文。

### N3. 流式体验

模型回复应尽快显示在历史对话区域。

流式输出过程中界面应保持可刷新、不卡死。

### N4. 终端兼容性

TUI 应尽量兼容 Windows PowerShell、Windows Terminal 和常见类 Unix 终端。

当终端能力不足时，不应直接崩溃，而应 fallback 到基础 REPL。

### N5. Provider 解耦

TUI 层不得直接依赖 OpenAI、Anthropic 或 DeepSeek 的具体请求格式。

TUI 层只通过已有 `ChatProvider` 抽象调用模型。

### N6. 会话一致性

TUI 模式和 REPL fallback 模式应使用同一套 `ChatSession` 会话模型。

同一进程内，用户消息和助手消息应按顺序保存。

### N7. 可维护性

TUI 布局、输入处理、消息渲染和 Provider 调用应尽量分离。

避免把所有逻辑堆在 `Main` 或单个巨型类中。

### N8. 依赖控制

本阶段只新增 TUI 必需依赖。

不引入完整 Agent、tool use、文件编辑或项目索引相关依赖。

### N9. 端到端可验收

开发完成后，应能在终端启动 CN Code，进入 TUI，输入真实问题，看到模型回复流式出现在历史对话区域，并能通过鼠标滚轮查看历史消息。

## 4. 不做的事

本阶段明确不实现以下能力：

1. 不实现 tool use。
2. 不读取、搜索、创建、修改或删除用户项目文件。
3. 不执行 shell 命令。
4. 不实现代码编辑、补丁生成或自动重构。
5. 不实现项目索引、代码库理解或代码导航。
6. 不持久化保存会话历史。
7. 不实现 token 用量统计。
8. 不实现耗时统计。
9. 不实现模型切换命令。
10. 不实现配置文件热重载。
11. 不实现输入历史上下键切换。
12. 不实现斜杠命令系统。
13. 不实现 Claude extended thinking 的可视化。
14. 不实现完整 Claude Code 级别的复杂布局，只做简洁 TUI 对话界面。

## 5. 验收标准

### A1. TUI 启动验收

执行 `cncode` 后，程序优先进入 TUI 对话界面。

界面包含历史对话区域、多行输入框和底部状态栏。

### A2. 状态栏验收

底部状态栏显示配置名、Provider 协议和模型名。

示例：

```text
default    openai/deepseek-v4-flash
```

### A3. 多行输入验收

用户在输入框输入多行内容时，Enter 插入换行。

用户按 Ctrl+Enter 后，当前输入被发送，输入框清空。

### A4. 流式回复验收

用户发送真实问题后，CN Code 调用配置的 Provider。

模型回复以流式方式逐字或逐块出现在历史对话区域。

回复不应等待完整生成后才显示。

### A5. 多轮上下文验收

用户先输入：“我叫小明，喜欢 Java。”

模型回复完成后，用户再输入：“我刚才说我喜欢什么语言？”

模型回复能参考前文，并指出用户喜欢 Java。

### A6. 鼠标滚动验收

当历史消息超过可视区域高度时，用户可以使用鼠标滚轮查看更早消息。

### A7. 自动滚动验收

模型流式回复过程中，历史对话区域自动滚动到底部，用户能看到最新生成内容。

### A8. 退出验收

用户发送完整内容 `exit` 或 `quit` 后，程序正常退出。

### A9. fallback 验收

当 TUI 初始化失败或终端不支持 TUI 时，程序打印清晰提示，并自动进入基础 REPL。

基础 REPL 仍能支持输入问题、流式回复、多轮上下文和退出命令。

### A10. 端到端验收

开发完成后，使用真实 DeepSeek 配置启动 CN Code。

在 TUI 中输入真实对话请求，观察 Provider 调用、流式输出、多轮上下文、鼠标滚动和退出行为是否符合本清单。
