# CN Code Web Markdown UI 规格说明

## 1. 背景与目标

CN Code 当前 Web UI 已经能展示对话、工具调用过程和模型流式输出，但界面偏工程化，消息内容按纯文本渲染，模型输出中的 Markdown 语法不会被格式化。用户希望前端界面接近 ChatGPT 的对话风格，并且助手输出按 Markdown 格式展示。

本阶段目标是在不改 Agent Loop、不改 Provider、不引入后端模板系统的前提下，改造 Web UI：

- 助手回复按 Markdown 渲染。
- 流式输出过程中逐步更新 Markdown。
- 界面风格接近 ChatGPT：居中对话栏、浅色背景、用户气泡、助手正文、底部圆角输入框。
- 工具事件以低调系统提示展示，不打断阅读。

## 2. 功能需求

### F1. Markdown 渲染

助手消息必须支持 Markdown 渲染。

至少支持：

- 标题
- 粗体
- 行内代码
- 代码块
- 列表
- 引用
- 链接
- 换行

### F2. 流式 Markdown 更新

模型流式返回 `delta` 时，前端累积原始 Markdown 文本，并重新渲染当前助手消息。

渲染过程中不能丢失文本，也不能把 Markdown 原文直接暴露成一坨纯文本。

### F3. 安全渲染

Markdown 渲染必须转义 HTML。

模型输出中的 `<script>`、HTML 标签或事件属性不得作为真实 HTML 执行。

### F4. ChatGPT 风格布局

Web UI 调整为类似 ChatGPT 的使用体验：

- 页面浅色背景。
- 左侧 sidebar 可保留，但视觉更轻。
- 主聊天区居中，最大宽度约 860px。
- 用户消息靠右，以浅色气泡显示。
- 助手消息靠左或占满内容栏，以正文形式显示。
- 输入框固定在底部，圆角、阴影、类似 ChatGPT 输入栏。

### F5. 工具事件展示

工具开始和工具结果仍可见。

工具事件使用低调系统行展示，例如：

```text
Tool read_file started
Tool read_file finished
```

工具事件不使用红色错误样式，除非工具失败。

### F6. 普通错误展示

错误事件仍明显展示。

错误消息使用红色或警告色。

### F7. 响应式

移动端隐藏 sidebar，聊天区和输入区保持可用。

文本、代码块和按钮不得溢出屏幕。

## 3. 非功能需求

### N1. 不引入构建工具

当前 Web UI 仍由 Java 字符串输出。

本阶段不引入 React、Vite、npm 或前端打包。

### N2. 不依赖外网 CDN

Markdown 渲染逻辑在 `WebAssets.java` 内完成。

不从 CDN 加载 marked.js、highlight.js 等外部库。

### N3. 性能可接受

每次 delta 重新渲染当前助手消息即可。

当前对话规模较小时无需虚拟列表。

### N4. 可维护

Markdown 渲染函数保持小型、可读。

不追求完整 CommonMark，只覆盖 Coding Agent 常用输出。

## 4. 不做的事

本阶段不实现：

1. 不实现完整 Markdown 标准。
2. 不实现代码高亮。
3. 不实现复制代码按钮。
4. 不实现主题切换。
5. 不实现消息编辑。
6. 不实现多会话列表。
7. 不修改 Agent Loop。
8. 不修改 Provider。

## 5. 验收标准

### A1. Markdown

助手输出 `# 标题` 时显示为标题。

助手输出 `**粗体**` 时显示为粗体。

助手输出 `` `code` `` 时显示为行内代码。

助手输出三反引号代码块时显示为代码块。

助手输出列表时显示为列表。

### A2. 安全

助手输出 `<script>alert(1)</script>` 时，页面显示文本，不执行脚本。

### A3. ChatGPT 风格

页面整体为浅色 ChatGPT 风格。

聊天内容居中，输入框在底部，用户消息为右侧气泡。

### A4. 工具事件

工具开始和结果事件仍可见。

工具成功不显示成红色错误。

工具失败显示为警告或错误。

### A5. 构建

`.\gradlew.bat check` 通过。

`.\gradlew.bat installDist` 通过。
