# CN Code 上下文压缩规格说明

## 1. 背景与目标

CN Code 已经具备多轮 Agent Loop、工具调用、权限检查、MCP 接入和 Web UI。随着工具系统变强，Token 消耗的大头会逐渐从用户消息转移到工具结果，尤其是读文件、搜代码、执行命令和 MCP 工具返回的大块内容。

本阶段目标是实现一套两层上下文压缩机制：

1. 请求前先做轻量预防，把过大的工具结果落盘，只把预览和路径留在对话里。
2. 当整体对话逼近上下文窗口时，再调用 LLM 生成结构化摘要替换旧历史。

压缩必须优先保护用户原始表达：用户消息尽量原文保留，不被摘要改写；工具结果和旧过程记录是主要压缩对象。

## 2. 功能需求

### F1. 工具结果轻量预防压缩

系统在每次 API 请求前，必须先检查对话历史里的工具结果。

单个工具结果超过阈值时，系统把完整内容写入会话目录中的磁盘文件，对话历史里只保留：

- 原始内容长度
- 预览文本
- 落盘文件路径
- 提示模型需要细节时重新读取该文件

同一条消息内所有工具结果合计超过阈值时，系统按内容大小从大到小依次落盘，直到合计大小回到阈值以内。

### F2. 工具结果替换状态

系统必须记录哪些工具结果已经处理过，避免每轮请求重复写盘或反复改变同一段历史。

已替换的工具结果在后续请求中必须保持字节稳定，不能因为再次压缩而生成不同预览文本。

### F3. 工具结果落盘

落盘目录必须位于当前项目或当前会话目录下，不能写到任意系统路径。

落盘文件名应与工具调用 ID 或稳定哈希关联，保证同一个工具结果可以稳定定位。

当模型需要完整工具结果时，它应通过读文件工具重新读取落盘文件，而不是要求系统把所有内容重新塞回上下文。

### F4. 整体上下文兜底压缩

系统在轻量预防压缩之后，必须估算当前对话总 token 消耗。

当估算值达到上下文窗口上限的阈值时，系统调用 LLM 生成结构化摘要，并用摘要替换较旧的消息历史。

摘要必须按固定部分组织，至少包含：

- 主要请求
- 关键概念
- 文件和代码
- 错误与修复
- 解决过程
- 用户原话
- 待办事项
- 当前工作状态
- 下一步

### F5. 摘要 Prompt 安全约束

摘要请求必须明确禁止 LLM 调用任何工具。

摘要 Prompt 的开头和结尾都必须强调：本次任务只允许总结上下文，禁止调用工具，禁止执行命令，禁止读写文件。

摘要 Prompt 必须要求模型先输出分析草稿，再输出正式摘要；系统只保留正式摘要，分析草稿用完即弃。

### F6. 压缩边界消息

完成整体压缩后，系统必须追加一条边界消息，提醒模型：

- 旧对话已经被摘要替换
- 摘要可能不包含完整文件细节
- 如果需要精确代码、命令输出或工具结果，必须重新调用工具读取
- 不要根据摘要脑补不存在的代码或文件内容

### F7. 手动压缩命令

用户可以通过命令手动触发压缩。

手动压缩必须跳过阈值判断，直接尝试生成结构化摘要。

手动压缩完成后，界面应显示压缩结果摘要，例如压缩前后估算 token 数。

### F8. 自动压缩熔断

自动压缩连续失败达到上限后，系统必须停止继续自动触发压缩，避免每轮请求都失败并进入死循环。

手动压缩不受自动熔断限制，但失败时必须给出清楚错误。

### F9. API 请求前固定顺序

每次 API 请求前必须按固定顺序执行：

1. 工具结果轻量预防压缩。
2. 整体上下文兜底压缩。
3. 构建最终要发送给 Provider 的消息。

轻量预防负责单条消息和工具结果体积，兜底压缩负责整体历史长度。

### F10. 用户原话保护

用户原始消息必须尽量保留原文。

整体压缩摘要中必须有“用户原话”部分，用于保留对后续任务仍有意义的关键用户表达。

系统不能把最新用户请求替换成摘要。

## 3. 非功能需求

### N1. 不污染主对话状态

轻量预防压缩应优先生成一份 API 请求视图，不直接破坏主 ConversationManager 中的完整历史。

如果需要持久记录替换状态，应单独存在状态对象或 JSONL 文件中。

### N2. 摘要调用禁止工具

用于摘要的 LLM 调用必须传空工具列表或禁用工具参数，确保摘要过程不会触发本地工具、MCP 工具或命令执行。

### N3. 失败可恢复

轻量预防压缩失败时，系统可以保留原工具结果继续请求，但必须避免崩溃。

整体压缩失败时，系统应保留原对话历史，并通过事件或日志报告失败原因。

### N4. 估算可接受

本阶段可以使用字符数近似估算 token，不要求接入真实 tokenizer。

估算逻辑必须集中封装，方便后续替换成真实 tokenizer。

### N5. 与现有 Agent Loop 兼容

压缩机制必须接入现有 `AgentLoop`，不改变用户发送消息、工具执行和流式输出的基本流程。

### N6. 与安全策略兼容

压缩落盘必须遵守路径沙箱。写入位置应固定在会话目录，不能由模型任意指定。

摘要调用不能绕过当前权限系统执行工具。

## 4. 设计骨架

新增上下文管理能力建议拆成两个包：

- `cncode.toolresult`：负责工具结果预算、落盘、预览、替换状态和记录恢复。
- `cncode.compact`：负责 token 估算、自动压缩、手动压缩、摘要 prompt、熔断状态和压缩边界消息。

主流程接入点在 `AgentLoop`：

1. 每轮调用 Provider 前，先生成轻量压缩后的 API 对话视图。
2. 再根据估算上下文窗口决定是否整体压缩。
3. 压缩成功后发送 `compact` 类事件给 Web UI。
4. Provider 调用使用压缩后的消息视图。

## 5. Out of Scope

- 不实现跨会话恢复完整上下文。
- 不实现真实 tokenizer。
- 不实现长期记忆系统。
- 不实现复杂分段摘要、多阶段摘要或树状摘要。
- 不改动工具权限策略本身。
- 不改变现有 Provider 协议字段。
- 不实现 SubAgent 或 Team 的上下文隔离压缩。
- 不做压缩过程的流式 UI 预览，只在压缩完成后显示结果。
